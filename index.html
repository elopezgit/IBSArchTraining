Introducción a la Arquitectura

Qué es, qué no es, y por qué importa en IBS

¿Qué es Arquitectura de Software?

Richards & Ford (Fundamentals of Software Architecture) la definen como la combinación de cuatro elementos inseparables:

🏗️

Estructura

El tipo de estilo(s) arquitectónico(s) del sistema. Ej: monolito modular, microservicios, event-driven.

⚡

Características Arquitectónicas

Los "-ilities": escalabilidad, disponibilidad, mantenibilidad, seguridad. Definen los criterios de éxito.

📋

Decisiones Arquitectónicas

Las reglas y restricciones que guían cómo se construye el sistema. Ej: "La UI no accede a la BD directamente".

🧩

Principios de Diseño

Guías (no reglas duras) para el equipo. Ej: "Preferir comunicación asíncrona entre módulos".

📖
Primera Ley de la Arquitectura (Richards & Ford): "Everything in software architecture is a trade-off." — Si crees que encontraste algo sin trade-off, es que aún no identificaste el trade-off.

Arquitectura ≠ Diseño

Es la confusión más común. La distinción importa porque determina quién toma la decisión:

Estilos Arquitectónicos

Monolito, Microservicios y Monolito Modular — por qué elegimos el tercero

Monolito Tradicional

Toda la aplicación como una sola unidad desplegable, con código fuertemente acoplado. Es el punto de partida de IBS actual (PowerBuilder + Sybase).

Microservicios

Servicios independientes, cada uno con su propia BD y ciclo de deploy. Máxima flexibilidad pero alta complejidad operativa.

⚠️
¿Por qué NO microservicios para IBS? La contabilidad bancaria requiere consistencia fuerte (ACID). Coordinar transacciones distribuidas entre docenas de microservicios introduce una complejidad enorme. Además, ~70% de la lógica está en stored procedures cross-domain, lo que hace el corte de servicios extremadamente difícil.

Monolito Modular ← Nuestra elección

Un punto medio pragmático: un solo proceso desplegable con módulos internos bien delimitados. Cada módulo tiene su propio dominio, interfaces claras y puede eventualmente extraerse como servicio.

Trade-offs Arquitectónicos

Toda decisión arquitectónica tiene un precio. El trabajo es conocer cuál.

Un trade-off es intercambiar una ventaja por otra. No existen decisiones gratuitas en arquitectura.

Ejemplo Real: ¿SAGAs o Transacciones Distribuidas?

✅ SAGA (elegido para IBS)

Ventajas: Alta disponibilidad, bajo acoplamiento, no bloquea recursos.

Costos: Consistencia eventual, necesita compensaciones, mayor complejidad de desarrollo.

❌ 2PC (Two-Phase Commit)

Ventajas: Consistencia ACID garantizada, más simple de razonar.

Costos: Bloqueo de recursos, baja disponibilidad si un servicio falla, no escala.

Framework de Análisis de Trade-offs

Para cualquier decisión, evaluar estas tres dimensiones (Head First SA):

🎯

¿Estratégico o Táctico?

Estratégico = largo plazo, muchos stakeholders, difícil de cambiar → Arquitectura

⚙️

¿Cuánto esfuerzo para cambiar?

Alto esfuerzo = decisión arquitectónica. Bajo esfuerzo = decisión de diseño.

⚖️

¿Trade-offs significativos?

Más trade-offs → más análisis requerido → más arquitectónica la decisión.

💡
Consejo práctico para IBS: Ante cada decisión técnica, documentarla como ADR (Architecture Decision Record) con: contexto, opciones consideradas, decisión tomada y trade-offs aceptados. Esto previene re-debates y aclara el razonamiento para nuevos integrantes.

Atributos de Calidad

Las "-ilities" que definen si el sistema es exitoso, más allá de la funcionalidad

Los atributos de calidad (también llamados non-functional requirements o architectural characteristics) son los criterios de éxito del sistema que no dependen de la funcionalidad. Para IBS, el Blueprint define:

⚡

Performance

Latencia y tiempo de respuesta. IBS mide p95/p99 por transacción crítica. Meta: <500ms en transacciones OLTP.

🔄

Disponibilidad

Uptime del sistema, generalmente en "nines" (99.9% = 8.7hs downtime/año). Crítico para operación bancaria.

📈

Escalabilidad

Mantener performance bajo carga creciente. Diferente de elasticidad (respuesta a bursts).

🔒

Seguridad

Autenticación, autorización, segregación de funciones, auditoría. Por diseño, no como parche.

🔧

Mantenibilidad

Facilidad de modificar el sistema sin efectos secundarios. Los bounded contexts del monolito modular la favorecen.

🌱

Flexibilidad / Agilidad

Capacidad de responder a cambios de negocio. Relacionada con testability y deployability.

Escalabilidad vs Elasticidad — Diferencia clave

📊 Escalabilidad

Performance consistente ante un número creciente y sostenido de usuarios. Ejemplo: el banco agrega 10.000 nuevos clientes.

⚡ Elasticidad

Resistencia ante bursts de tráfico súbito. Ejemplo: cierre mensual, pago de haberes — todos los cajeros a la vez.

ℹ️
Regla de oro: No se pueden optimizar todos los atributos simultáneamente. Elegir las 3 más importantes para el sistema y dejar que guíen las decisiones de diseño. Para IBS: Disponibilidad · Seguridad/Auditoría · Mantenibilidad.

El estilo arquitectónico elegido para el nuevo IBS

Definición

Un sistema que se despliega como una unidad única pero cuyo código se organiza en módulos con límites claros, reglas de dependencia explícitas y ownership definido.

Principios Clave

📦 Alta Cohesión

Todo lo relacionado con un dominio (Pagos, Clientes, Créditos) vive en el mismo módulo. Los cambios de negocio son locales.

🔗 Bajo Acoplamiento

Los módulos se comunican solo a través de interfaces (fachadas) bien definidas. Nunca acceden directamente a tablas de otro módulo.

🚪 Encapsulamiento Fuerte

Nada accede al interior de un módulo sino a través de sus contratos. La BD de cada módulo es privada.

🌱 Evolución Controlada

Los módulos pueden extraerse como microservicios en el futuro si hay necesidad de escalar. El diseño lo permite sin big-bang.

Patrones de Organización del Código

Existen tres formas de organizar código. El Monolito Modular favorece la tercera:

🗂️ Por Capa (legacy)

src/
  controllers/
  services/
  repositories/
  models/

❌ Dificulta el trabajo en features completas

🏷️ Por Feature

src/
  customer/
  payments/
  accounts/
  loans/

⚡ Mejor pero sin límites de dominio claros

🧩 Por Módulo/Componente

src/
  modules/
    payments/
      domain/
      app/
      infra/
    accounts/
      ...

✅ Límites explícitos, cada módulo independiente

Capas y Encapsulamiento

Organización interna de cada módulo

Arquitectura Interna de un Módulo

Dentro de cada módulo, se aplica la Arquitectura Hexagonal (o Clean Architecture). El núcleo del dominio no conoce los detalles de infraestructura.

🖥️ Capa de Presentación / API Controllers · BFF · DTOs de entrada
▼ depende de ▼
⚙️ Capa de Aplicación Use Cases · Application Services · DTOs
▼ depende de ▼
💎 Capa de Dominio (núcleo) Entities · Aggregates · Domain Services · Events
▼ define interfaces, implementadas en ▼
🗄️ Capa de Infraestructura Repositorios · ORM · Mensajería · APIs externas
💡
Regla de Dependencia: Las capas internas nunca conocen a las externas. El Dominio NO importa de Infraestructura. Si necesitas la BD desde el Dominio, defines una interfaz (IAccountRepository) y la Infraestructura la implementa. Esto se logra con Inyección de Dependencias.

Ejemplo en .NET para el módulo Pagos

// DOMINIO — No sabe nada de EF, SQL ni infraestructura
public class Transferencia
{
    public TransferenciaId Id { get; private set; }
    public Money Monto { get; private set; }
    public CuentaId CuentaOrigen { get; private set; }

    public Result Ejecutar(Cuenta origen, Cuenta destino)
    {
        if (origen.Saldo < Monto)
            return Result.Failure("Saldo insuficiente");
        origen.Debitar(Monto);
        destino.Acreditar(Monto);
        AddDomainEvent(new TransferenciaEjecutadaEvent(Id));
        return Result.Success();
    }
}

// APLICACIÓN — Orquesta el flujo
public class EjecutarTransferenciaUseCase
{
    private readonly ICuentaRepository _cuentas;  // interfaz, no implementación
    private readonly IEventPublisher _events;

    public async Task<Result> Execute(TransferirDto cmd)
    {
        var origen = await _cuentas.GetById(cmd.CuentaOrigenId);
        var destino = await _cuentas.GetById(cmd.CuentaDestinoId);
        var transferencia = new Transferencia(cmd.Monto);
        var result = transferencia.Ejecutar(origen, destino);
        if (result.IsSuccess())
            await _cuentas.Save(origen, destino, transferencia);
        return result;
    }
}

Inyección de Dependencias

El mecanismo que hace posible el bajo acoplamiento

La Inyección de Dependencias (DI) es el patrón que permite que las clases reciban sus dependencias desde afuera en lugar de crearlas internamente. Es el mecanismo central del Principio de Inversión de Dependencias (la "D" de SOLID).

Problema sin DI (acoplamiento duro)

public class PagoService
{
    // ❌ PagoService crea sus dependencias — acoplamiento duro
    private readonly SqlCuentaRepository _repo = new SqlCuentaRepository();

    // No podemos testear sin BD real
    // No podemos cambiar la implementación sin tocar PagoService
}

Solución con DI (inversión de dependencias)

// 1. Definir la abstracción (en capa de Dominio)
public interface ICuentaRepository
{
    Task<Cuenta> GetById(CuentaId id);
    Task Save(Cuenta cuenta);
}

// 2. Implementación en Infraestructura
public class EfCuentaRepository : ICuentaRepository
{
    private readonly IbsDbContext _db;
    public EfCuentaRepository(IbsDbContext db) => _db = db;
    public async Task<Cuenta> GetById(CuentaId id)
        => await _db.Cuentas.FindAsync(id.Value);
}

// 3. PagoService recibe la abstracción — no sabe qué implementación
public class PagoService
{
    private readonly ICuentaRepository _repo;  // ✅ interfaz
    public PagoService(ICuentaRepository repo) => _repo = repo;
}

// 4. Registrar en el contenedor DI (Program.cs)
builder.Services.AddScoped<ICuentaRepository, EfCuentaRepository>();
// Para tests: AddScoped<ICuentaRepository, InMemoryCuentaRepository>()
💡
Beneficio clave para IBS: La migración de stored procedures a .NET se hace módulo por módulo. Con DI, el módulo "Pagos" puede tener una implementación temporal que llama al SP (SqlPaymentRepository) y luego reemplazarla por la implementación .NET pura (EfPaymentRepository), sin tocar el código de dominio ni aplicación.

DDD Práctico

Domain-Driven Design — el lenguaje para modelar el dominio bancario

DDD (Eric Evans, 2003) es un enfoque de diseño que pone el modelo de dominio en el centro. El código habla el mismo idioma que el negocio.

Bloques de Construcción

🏛️ Entity (Entidad)

Objeto con identidad única que persiste en el tiempo. Ej: Cuenta, Cliente, Transferencia. Dos cuentas con distinto ID son objetos distintos aunque tengan el mismo saldo.

💎 Value Object

Objeto definido por sus atributos, sin identidad. Son inmutables. Ej: Money, IBAN, Periodo. Dos Money(100, ARS) son iguales e intercambiables.

🫙 Aggregate

Grupo de entidades que forman una unidad de consistencia. Solo se accede desde la raíz (Aggregate Root). Ej: CuentaCorriente con sus movimientos.

⚙️ Domain Service

Operaciones de dominio que no pertenecen naturalmente a una entidad. Ej: CalculadoraIntereses, ValidadorLimitePagos.

📡 Domain Event

Algo que ocurrió en el dominio y otros módulos pueden necesitar saber. Ej: CuentaAbierta, TransferenciaEjecutada, PrestamoAprobado.

🗄️ Repository

Interfaz para persistir y recuperar Aggregates. El Dominio define qué necesita; la Infraestructura implementa cómo.

Ejemplo completo: Módulo Cuentas

// VALUE OBJECT — inmutable, sin identidad
public record Money(decimal Amount, Currency Currency)
{
    public Money Add(Money other)
    {
        if (Currency != other.Currency) throw new InvalidOperationException();
        return new Money(Amount + other.Amount, Currency);
    }
    public static bool operator >=(Money a, Money b) => a.Amount >= b.Amount;
}

// AGGREGATE ROOT — unidad de consistencia
public class CuentaCorriente : AggregateRoot<CuentaId>
{
    public ClienteId Titular { get; private set; }
    public Money Saldo { get; private set; }
    private List<Movimiento> _movimientos = new();

    // Lógica de negocio dentro del aggregate
    public void Debitar(Money monto, string concepto)
    {
        if (Saldo < monto)
            throw new SaldoInsuficienteException(Id, monto);
        Saldo = Saldo - monto;
        _movimientos.Add(Movimiento.Debito(monto, concepto));
        AddDomainEvent(new SaldoActualizadoEvent(Id, Saldo));
    }
}

// DOMAIN SERVICE — opera sobre múltiples aggregates
public class ServicioTransferencia
{
    public Result Transferir(CuentaCorriente origen, 
                              CuentaCorriente destino, 
                              Money monto)
    {
        origen.Debitar(monto, "Transferencia saliente");
        destino.Acreditar(monto, "Transferencia entrante");
        return Result.Success();
    }
}

Límites entre Módulos — Bounded Contexts

La decisión más difícil y más importante del diseño

Un Bounded Context (DDD) es el límite conceptual dentro del cual un modelo de dominio es consistente. Es la guía para decidir qué va en qué módulo.

⚠️
El mayor desafío del Monolito Modular (Blueprint IBS): Los límites mal definidos crean acoplamiento oculto que es peor que el original. Una vez que los módulos están en producción, cambiar sus límites es muy costoso.

Cómo definir los límites (criterios)

1. Ubiquitous Language

El término "Cuenta" tiene distinto significado en Contabilidad (cuenta contable) y en Productos (cuenta bancaria del cliente). Si el lenguaje diverge → distinto bounded context.

2. Cohesión Funcional

¿Qué cosas cambian juntas? Si agregar un nuevo producto bancario requiere cambiar 5 módulos distintos, los límites están mal trazados.

3. Ownership de Datos

Cada tabla/entidad tiene un único dueño. Si dos módulos escriben en la misma tabla → hay un problema de límites.

4. Equipos Autónomos

Un módulo debe poder ser desarrollado por un equipo sin coordinar constantemente con otros. Si hay coordinación permanente → los límites están mal.

Integración Síncrona entre Módulos

Cuándo usarla, cuándo no, y cómo implementarla

¿Qué es una llamada síncrona?

El módulo llamador espera la respuesta antes de continuar. El flujo es bloqueante.

Flujo síncrono — Módulo Pagos consulta Clientes
Módulo Pagos          Módulo Clientes
     │                      │
     │──── ValidarCliente ──►│
     │       (bloquea)       │ consulta BD
     │                      │ valida KYC/AML
     │◄─── Result{ok/error}──│
     │                       │
     │ continúa con el pago   │

Implementación: Fachadas de Módulo en .NET

// En módulo Clientes — define su contrato público
public interface IClientesFacade
{
    Task<ClienteDto> GetCliente(ClienteId id);
    Task<bool> ValidarKyc(ClienteId id);
}

// En módulo Pagos — usa la fachada, no sabe nada interno de Clientes
public class ProcesarPagoUseCase
{
    private readonly IClientesFacade _clientes;

    public async Task<Result> Execute(OrdenPagoCmd cmd)
    {
        // Llamada síncrona — espera validación
        var kycOk = await _clientes.ValidarKyc(cmd.ClienteId);
        if (!kycOk) return Result.Failure("Cliente no validado KYC");
        // ... continúa el pago
    }
}

¿Cuándo usar llamadas síncronas?

✅ Úsala cuando:

• Necesitás la respuesta para continuar
• Es una validación de negocio (KYC, límites)
• El tiempo de espera es mínimo y predecible
• La consistencia inmediata es requerida

❌ Evítala cuando:

• El módulo destino puede estar lento o caído
• La operación puede hacerse después
• Genera acoplamiento temporal
• Impacta en escalabilidad

Integración Asíncrona — Colas y Eventos

Event-Driven Architecture dentro del monolito modular

Comunicación Asíncrona

El módulo publicador no espera respuesta. Publica un mensaje/evento y continúa. Los suscriptores reaccionan cuando pueden.

Event-Driven — Transferencia ejecutada notifica Contabilidad
Módulo Pagos          Event Bus          Módulo Contabilidad
     │                   │                      │
     │ ejecuta pago       │                      │
     │──TransferenciaEjec►│                      │
     │ (continúa)         │                      │
     │                    │──TransferenciaEjec──►│
     │                    │                      │ registra asientos
     │                    │                      │ GL independiente

Tipos de Comunicación Asíncrona

📨 Cola (Queue)

Point-to-point. Un productor, un consumidor. Cada mensaje es procesado por exactamente un receptor. Ideal para comandos: "Procesar esta transferencia".

📡 Topic / Event

Pub/Sub. Un productor, múltiples consumidores. Todos los interesados reciben la copia. Ideal para eventos: "TransferenciaEjecutada" → notifica Contabilidad, Notificaciones, Reportes.

Implementación en .NET (MediatR + eventos de dominio)

// 1. Definir el Domain Event
public record TransferenciaEjecutadaEvent(
    TransferenciaId Id,
    Money Monto,
    CuentaId Origen,
    CuentaId Destino,

Idempotencia

Ejecutar la misma operación N veces debe tener el mismo efecto que ejecutarla una vez

En sistemas distribuidos y bancarios, los reintentos son inevitables (timeouts, caídas de red, errores transitorios). La idempotencia garantiza que reintentar no genera duplicados.

🚨
Escenario crítico en IBS: Un pago se envía, la red falla antes de recibir respuesta. El cliente reintenta. Si el sistema no es idempotente → el cliente paga dos veces. Esto es un error regulatorio grave en un sistema bancario.

Implementación con Idempotency-Key

// 1. El cliente envía un idempotency-key único por operación
POST /api/pagos/transferencias
Headers:
  Idempotency-Key: "client-uuid-550e8400"
Body:
  { "monto": 1000, "destino": "CR-001" }

// 2. El endpoint verifica si ya fue procesado
public class TransferenciasController
{
    public async Task<IActionResult> Post(
        [FromHeader] string idempotencyKey,
        TransferirRequest request)
    {
        // Buscar si ya existe resultado para esta key
        var cached = await _idempotency.TryGetResult(idempotencyKey);
        if (cached != null) return Ok(cached);   // devuelve resultado previo

        // Primera vez — procesar
        var result = await _useCase.Execute(request);

        // Guardar resultado para futuros reintentos
        await _idempotency.SaveResult(idempotencyKey, result, TimeSpan.FromHours(24));
        return Ok(result);
    }
}

// 3. Tabla de idempotencia (persiste en misma BD)
CREATE TABLE idempotency_keys (
    key        VARCHAR(100) PRIMARY KEY,
    result     JSONB,
    created_at TIMESTAMP,
    expires_at TIMESTAMP
);
💡
Dónde aplica en IBS: Toda operación de escritura que pueda reintentarse debe ser idempotente: transferencias, pagos, apertura de cuentas, posteos contables. Los SAGAs también requieren idempotencia en cada paso para manejar reintentos ante fallas parciales.

Patrón Outbox

Garantizar la publicación confiable de eventos junto con cambios de datos

El Problema

Guardar en la BD y publicar un evento son dos operaciones separadas. Entre ellas puede fallar el proceso.

Problema sin Outbox — Riesgo de pérdida de eventos
                        ❌ PROCESO FALLA AQUÍ ❌
                                │
Módulo Pagos ──► Guardar en BD ─┤─► Publicar evento
                   (exitoso)    │   (nunca ocurre)
                                │
Resultado: BD actualizada, Contabilidad no se enteró

La Solución: Transactional Outbox

Persistir el evento en la misma transacción que los datos, en una tabla outbox. Un proceso separado (Relay) publica los eventos pendientes.

Patrón Outbox — Garantía at-least-once delivery
 STEP 1: Una sola transacción atómica
 ┌────────────────────────────────────────────┐
 │  BEGIN TRANSACTION                        │
 │    UPDATE cuentas SET saldo = saldo - 100  │
 │    INSERT INTO outbox                      │
 │      (type='TransferenciaEjecutada',       │
 │       payload='{...}', published=false)    │
 │  COMMIT TRANSACTION                        │
 └────────────────────────────────────────────┘

 STEP 2: Proceso Relay (background)
 ┌────────────────────────────────────────────┐
 │  SELECT * FROM outbox                      │
 │  WHERE published = false                   │
 │  ORDER BY created_at                       │
 │                                            │
 │  FOR EACH evento:                          │
 │    → Publicar al Event Bus                 │
 │    → UPDATE outbox SET published=true      │
 └────────────────────────────────────────────┘

Implementación en .NET

// 1. Tabla outbox (misma BD que el módulo)
public class OutboxMessage
{
    public Guid Id { get; set; } = Guid.NewGuid();
    public string Type { get; set; }       // "TransferenciaEjecutada"
    public string Payload { get; set; }    // JSON serializado
    public bool Published { get; set; }
    public DateTime CreatedAt { get; set; }
}

// 2. En el Application Service — guardar dato + outbox en una transacción
public class EjecutarTransferenciaHandler
{
    public async Task<Result> Handle(TransferirCmd cmd)
    {
        await using var tx = await _db.BeginTransactionAsync();
        try
        {
            // Operación de negocio
            var transferencia = await _service.Ejecutar(cmd);

            // Guardar evento en outbox — MISMA transacción
            _db.OutboxMessages.Add(new OutboxMessage
            {
                Type = "TransferenciaEjecutada",
                Payload = JsonSerializer.Serialize(transferencia),
            });

            await _db.SaveChangesAsync();
            await tx.CommitAsync();
            return Result.Success();
        }
        catch
        {
            await tx.RollbackAsync();
            throw;
        }
    }
}

// 3. Relay — background service que publica lo pendiente
public class OutboxRelayService : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        while (!ct.IsCancellationRequested())
        {
            var pending = await _db.OutboxMessages
                .Where(m => !m.Published).ToListAsync();
            foreach (var msg in pending)
            {
                await _eventBus.Publish(msg.Type, msg.Payload);
                msg.Published = true;
            }
            await _db.SaveChangesAsync();
            await Task.Delay(1000, ct);
        }
    }
}

Patrón SAGA

Consistencia en transacciones que tocan múltiples módulos

El Problema: Transacciones Cross-Módulo

En IBS, una transferencia toca Pagos, Cuentas y Contabilidad. No podemos usar una sola transacción ACID porque cada módulo tiene su propia BD (o schema). Necesitamos coordinar una secuencia de transacciones locales con capacidad de deshacer los pasos anteriores si algo falla.

¿Qué es una SAGA?

Una secuencia de transacciones locales, donde cada paso publica un evento/mensaje que dispara el siguiente. Si un paso falla, se ejecutan transacciones compensatorias para deshacer los pasos anteriores.

SAGA: Flujo de Transferencia — Camino Feliz
  API Gateway
      │
      ▼
  ┌─────────────────────────────────────────────────────────────────┐
  │                  SAGA ORQUESTADOR                               │
  │                                                                 │
  │  PASO 1: Validar Cliente (Módulo Clientes)                      │
  │     ──────────────────────────────► ok                         │
  │                                                                 │
  │  PASO 2: Reservar Saldo (Módulo Cuentas)                        │
  │     ──────────────────────────────► ok (saldo reservado)       │
  │                                                                 │
  │  PASO 3: Registrar Pago (Módulo Pagos)                          │
  │     ──────────────────────────────► ok                         │
  │                                                                 │
  │  PASO 4: Postear GL (Módulo Contabilidad)                       │
  │     ──────────────────────────────► ok                         │
  │                                                                 │
  │  PASO 5: Confirmar Saldo (Módulo Cuentas)                       │
  │     ──────────────────────────────► ok ✅                      │
  └─────────────────────────────────────────────────────────────────┘
SAGA: Flujo con Falla — Compensaciones
  PASO 1: Validar Cliente ──────────────────► ok
  PASO 2: Reservar Saldo  ──────────────────► ok (saldo reservado)
  PASO 3: Registrar Pago  ──────────────────► ok
  PASO 4: Postear GL      ──────── FALLA ❌

  COMPENSACIONES (en orden inverso):
  COMP-3: Cancelar Pago   ──────────────────► ok
  COMP-2: Liberar Saldo   ──────────────────► ok (saldo liberado)
  COMP-1: (no necesaria)
  
  Estado final: Todo revertido, cliente notificado del fallo

Orquestación vs Coreografía

🎼 Orquestación (elegido IBS)

Un componente central (Saga Orchestrator) coordina los pasos y toma decisiones. Más fácil de entender, testear y monitorear.

SagaOrchestrator
  ├── paso1 → Clientes
  ├── paso2 → Cuentas
  └── paso3 → Pagos

💃 Coreografía

Cada servicio reacciona a eventos de los demás. No hay coordinador central. Más flexible pero más difícil de trazar y debuggear.

Cuentas emite → Pagos escucha
Pagos emite → Contabilidad escucha
(emerge la coordinación)

Implementación básica en .NET

public class TransferenciaSaga
{
    private readonly IClientesFacade _clientes;
    private readonly ICuentasFacade _cuentas;
    private readonly IPagosFacade _pagos;
    private readonly ISagaStateRepository _state;

    public async Task<SagaResult> Execute(TransferenciaCmd cmd)
    {
        var sagaId = Guid.NewGuid();
        await _state.Save(sagaId, "Iniciada");

        // PASO 1
        var kycOk = await _clientes.ValidarKyc(cmd.ClienteId);
        if (!kycOk) return SagaResult.Failure("KYC inválido");

        // PASO 2
        var reserva = await _cuentas.ReservarSaldo(cmd.CuentaOrigen, cmd.Monto);
        if (!reserva.Ok) return SagaResult.Failure("Saldo insuficiente");

        try
        {
            // PASO 3
            await _pagos.RegistrarPago(cmd, reserva.Id);
            // PASO 4 (postear GL via evento async)
            // PASO 5
            await _cuentas.ConfirmarDebito(reserva.Id);
            await _state.Save(sagaId, "Completada");
            return SagaResult.Success();
        }
        catch (Exception ex)
        {
            // COMPENSACIONES
            await _pagos.CancelarPago(reserva.Id);
            await _cuentas.LiberarReserva(reserva.Id);
            await _state.Save(sagaId, "Compensada");
            return SagaResult.Failure(ex.Message);
        }
    }
}
⚠️
Requisito crítico: Cada paso de la SAGA debe ser idempotente. Si el proceso falla a mitad de una compensación, debe poder reintentarse sin duplicar efectos. Por eso se guardan los IDs de reserva y se usa el patrón Outbox para los eventos que disparan cada paso.

Observabilidad

Ver qué pasa adentro del sistema sin necesidad de reiniciarlo

Observabilidad = capacidad de entender el estado interno de un sistema a partir de sus outputs externos. Los tres pilares son:

📝

Logs Estructurados

Logs en formato JSON (no texto libre). Permiten búsqueda, filtrado y agregación en herramientas como Elasticsearch o Splunk.

🔍

Trazabilidad (Tracing)

Seguir una transacción de punta a punta a través de todos los módulos. Requiere propagar un Correlation ID.

📊

Métricas

Datos numéricos en el tiempo: latencia p95/p99, tasa de errores, throughput, uso de memoria. Expuestos via Prometheus/Grafana.

Correlation ID — La clave de la trazabilidad

Un identificador único que se genera en la entrada del sistema y se propaga a todos los módulos, logs y eventos de esa transacción.

// Middleware que asegura que toda request tenga correlation ID
public class CorrelationIdMiddleware
{
    public async Task InvokeAsync(HttpContext context)
    {
        // Tomar del header o generar uno nuevo
        var correlationId = context.Request.Headers["X-Correlation-ID"]
            .FirstOrDefault() ?? Guid.NewGuid().ToString();

        // Disponible en toda la request
        context.Items["CorrelationId"] = correlationId;
        context.Response.Headers["X-Correlation-ID"] = correlationId;

        // Propagar en todos los logs de esta request
        using (LogContext.PushProperty("CorrelationId", correlationId))
        {
            await _next(context);
        }
    }
}

// Log estructurado — todos llevan CorrelationId

Principios de Seguridad por Diseño (Blueprint IBS)

🔍 Auditoría Inmutable

Toda operación sensible se registra en una tabla append-only. Nadie puede modificar el historial de operaciones. Requerimiento regulatorio bancario.

🔒 Segregación de Funciones

Quien aprueba no es quien ejecuta. Quien puede ver datos de clientes no puede procesar pagos. Previene fraude interno.

🛡️ DLP (Data Loss Prevention)

Datos sensibles (PII, números de cuenta) se tokenizan o cifran. Nadie accede al dato real salvo el módulo que lo gestiona.

🌐 Zero Trust en APIs

Toda llamada entre módulos debe autenticarse, incluso si vienen de dentro del mismo proceso. Un módulo comprometido no compromete a otros.

Estrategia y Gestión

Big Bang Migration
Enfoque de "todo o nada" descartado por su alto riesgo. Se prefiere la migración por oleadas (Waves).
Drivers de Esfuerzo
Factores que determinan el costo del proyecto, como la cantidad de SPs a migrar y la complejidad de los flujos multi-dominio.
Modernización Evolutiva
Estrategia de cambiar el sistema por partes mientras sigue operando, evitando interrupciones al negocio bancario.
Patrón Strangler (Estrangulador)
Metodología donde las nuevas funcionalidades en .NET van "envolviendo" al legado hasta que este desaparece.
Wave (Oleada)
Conjunto de releases que agrupan módulos funcionales (ej. Wave 1: Préstamos, Wave 2: Depósitos) según el plan 2026.
WBS (Work Breakdown Structure)
Descomposición del proyecto en tareas manejables: Gestión, Arquitectura, Plataforma, Datos, Migración y Testing.

Arquitectura (Monolito Modular)

ADR (Architecture Decision Record)
Registro formal de una decisión técnica para evitar "amnesia colectiva" y re-debates cíclicos.
Anticorruption Layer (ACL)
Capa que traduce datos del sistema viejo al nuevo, impidiendo que el diseño sucio del legado afecte al código limpio.
Bounded Context
Límite donde un modelo de datos es consistente. Un módulo de "Clientes" no es igual al de "Riesgos".
Modularidad Fuerte
Capacidad de cambiar un módulo internamente sin afectar a los demás, lograda mediante interfaces claras.
Quiescence
Estado de reposo necesario para migrar datos de un módulo sin que haya transacciones activas.
Trade-off
Decisión donde se gana algo (ej. escalabilidad) perdiendo otra cosa (ej. simplicidad). No hay soluciones perfectas.

Datos e Integración

ACID
Propiedad no negociable para el Core Bancario: Atomicidad, Consistencia, Aislamiento y Durabilidad.
Compensación (SAGA)
Acción de reversa que se ejecuta si un paso intermedio de una transacción distribuida falla.
Consistencia Eventual
Modelo donde los datos se sincronizan tras un breve lapso de tiempo, aceptable en flujos no contables.
Idempotencia
Garantía de que procesar el mismo mensaje dos veces no duplique el efecto (ej. no cobrar dos veces el mismo interés).
Outbox Pattern
Técnica para asegurar que los eventos se guarden en la DB antes de enviarse a una cola, evitando pérdida de datos.
SAGA Pattern
Gestión de transacciones largas que abarcan múltiples módulos sin usar bloqueos pesados de base de datos.
Schema Firewall
Reglas y contratos para evitar que aplicaciones externas o procesos legados corrompan el nuevo esquema de datos.

Legado y Tecnología

Coexistencia
Periodo crítico donde PB y .NET operan sobre la misma base de datos Sybase simultáneamente.
Deuda Técnica
Costo acumulado por años de parches en PowerBuilder que dificultan el mantenimiento actual.
Lógica In-DB
Código de negocio escrito directamente en Stored Procedures (SPs), el cual se busca extraer hacia servicios C#.
PowerBuilder (PB)
Lenguaje del sistema IBS actual. Su obsolescencia es el driver principal del programa de modernización.
Sybase ASE
Motor de base de datos actual. La estrategia busca optimizar el acceso desde la nueva arquitectura .NET.

Métricas de Éxito (KPIs)

Defect Leakage
Porcentaje de fallos que llegan a manos del usuario. Meta: < 5% por cada release.
MTTR (Mean Time To Repair)
Tiempo promedio de resolución de incidentes críticos (P1/P2) detectados durante el piloto.
P95 Latency
El tiempo de respuesta que no es superado por el 95% de las transacciones. Mide la velocidad real percibida.
Release Readiness Score
Criterios de salida que incluyen cobertura de tests (>80%), seguridad y performance aprobados.
Tasa de Compensación
Porcentaje de SAGAs que fallan y requieren reversa. Un número alto indica inestabilidad en el diseño.
