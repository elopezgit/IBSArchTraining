Idempotencia: Blindaje contra Duplicados

Garantizando que una operación financiera ocurra exactamente una vez.

En el mundo real, los cables se cortan y los servidores se reinician. La Idempotencia es la propiedad de una operación que permite que sea llamada múltiples veces sin cambiar el resultado más allá de la primera aplicación. En IBS, es nuestra defensa contra el "Doble Gasto".

El Problema: La Falacia de la Red Confiable

Imagina este flujo en un pago móvil:

  1. 1. El usuario pulsa "Pagar". La App envía la petición.
  2. 2. El servidor procesa el pago con éxito.
  3. 3. Falla la red: La respuesta "OK" nunca llega a la App.
  4. 4. El usuario, impaciente, pulsa "Pagar" de nuevo.

Estrategias de Implementación

🔑 Client-Side Keys (Tokens)

El cliente genera un UUID único para la transacción. El servidor guarda este UUID y el resultado. Si recibe el mismo UUID, devuelve el resultado guardado sin procesar de nuevo.

🛡️ Natural Keys (Llaves de Negocio)

Usamos datos del negocio para detectar duplicados.
Ejemplo: No permitir dos transferencias del mismo Origen al mismo Destino con el mismo Monto en el mismo Minuto.

Implementación Técnica en .NET 8

Para que la idempotencia sea infalible en un banco, el chequeo de la llave y la operación deben ser Atómicos (ocurrir en la misma transacción de base de datos).

public async Task<Result> HandleTransferencia(Request req, string key) {
    // Empezamos transacción SQL
    using var transaction = await _unitOfWork.BeginTransactionAsync();

    // 1. Intentar insertar la llave (Unique Constraint en DB)
    var isNew = await _db.IdempotencyKeys.AddIfNotExists(key);
    
    if (!isNew) {
        // 2. Si ya existe, retornar el resultado previo (Cache o DB)
        return await _db.IdempotencyKeys.GetPreviousResult(key);
    }

    // 3. Si es nueva, ejecutar lógica de negocio
    var result = await _logic.Execute(req);

    // 4. Guardar resultado y confirmar
    await _db.IdempotencyKeys.UpdateResult(key, result);
    await transaction.CommitAsync();
    
    return result;
}

Diferencia entre Idempotencia y Lectura

Método ¿Es Idempotente? Efecto Colateral
GET (Consultar Saldo) Sí (Nativo) Ninguno. Solo lectura.
DELETE (Eliminar Beneficiario) Sí Borrar algo que ya no existe da el mismo resultado.
POST (Crear Pago) No (Requiere lógica) Cada llamada crearía un registro nuevo si no se controla.
💡
Concepto de Nivelación: El "Manejo de Reintentos". La idempotencia permite que nuestras políticas de reintentos (Polly) sean agresivas. Si la red falla, podemos reintentar 3 veces sin miedo, sabiendo que el servidor es lo suficientemente inteligente para ignorar los duplicados.

Regla de Oro en IBS:

"Toda mutación de saldo DEBE ser idempotente". No confíes en que el cliente enviará la petición una sola vez. Diseña el servidor asumiendo que recibirá la misma petición múltiples veces debido a errores de red.
