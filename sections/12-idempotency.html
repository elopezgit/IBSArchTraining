Idempotencia

Ejecutar la misma operación N veces debe tener el mismo efecto que ejecutarla una vez

En sistemas distribuidos y bancarios, los reintentos son inevitables (timeouts, caídas de red, errores transitorios). La idempotencia garantiza que reintentar no genera duplicados.

🚨
Escenario crítico en IBS: Un pago se envía, la red falla antes de recibir respuesta. El cliente reintenta. Si el sistema no es idempotente → el cliente paga dos veces. Esto es un error regulatorio grave en un sistema bancario.

Implementación con Idempotency-Key

// 1. El cliente envía un idempotency-key único por operación
POST /api/pagos/transferencias
Headers:
  Idempotency-Key: "client-uuid-550e8400"
Body:
  { "monto": 1000, "destino": "CR-001" }

// 2. El endpoint verifica si ya fue procesado
public class TransferenciasController
{
    public async Task<IActionResult> Post(
        [FromHeader] string idempotencyKey,
        TransferirRequest request)
    {
        // Buscar si ya existe resultado para esta key
        var cached = await _idempotency.TryGetResult(idempotencyKey);
        if (cached != null) return Ok(cached);   // devuelve resultado previo

        // Primera vez — procesar
        var result = await _useCase.Execute(request);

        // Guardar resultado para futuros reintentos
        await _idempotency.SaveResult(idempotencyKey, result, TimeSpan.FromHours(24));
        return Ok(result);
    }
}

// 3. Tabla de idempotencia (persiste en misma BD)
CREATE TABLE idempotency_keys (
    key        VARCHAR(100) PRIMARY KEY,
    result     JSONB,
    created_at TIMESTAMP,
    expires_at TIMESTAMP
);
💡
Dónde aplica en IBS: Toda operación de escritura que pueda reintentarse debe ser idempotente: transferencias, pagos, apertura de cuentas, posteos contables. Los SAGAs también requieren idempotencia en cada paso para manejar reintentos ante fallas parciales.
