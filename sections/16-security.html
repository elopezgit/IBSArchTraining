Seguridad

Autenticación, Autorización y Seguridad en APIs

🔐

Autenticación (AuthN)

¿Quién sos? Verificar la identidad del usuario o sistema. En IBS: OIDC (OpenID Connect) sobre OAuth 2.0. El usuario se autentica contra un Identity Provider y recibe un JWT.

🛡️

Autorización (AuthZ)

¿Qué podés hacer? Verificar que el usuario autenticado tiene permiso para la operación. En IBS: RBAC (Role-Based Access Control) + segregación de funciones.

Flujo JWT en APIs

Flujo de Autenticación con JWT
 Cliente/App         Identity Provider        API IBS
     │                      │                   │
     │── Login (usr/pwd) ──►│                   │
     │                      │ valida credencial  │
     │◄── JWT Token ────────│                   │
     │   {                  │                   │
     │    "sub": "user-123" │                   │
     │    "roles": ["cajero"]│                  │
     │    "exp": 1714000000 │                   │
     │   }                  │                   │
     │                      │                   │
     │─── POST /transferencias ─────────────────►│
     │    Authorization: Bearer {JWT}            │
     │                                           │ valida firma JWT
     │                                           │ verifica roles
     │◄────────────────────── 200 OK ────────────│

Implementación en .NET

// Registrar autenticación JWT en Program.cs
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.Authority = "https://identity.ibs.bank";
        options.Audience = "ibs-api";
    });

builder.Services.AddAuthorization(options =>
{
    // Políticas basadas en roles — segregación de funciones
    options.AddPolicy("PuedeProcesarPagos", p =>
        p.RequireRole("cajero", "supervisor"));

    options.AddPolicy("PuedeAprobarCreditos", p =>
        p.RequireRole("analista-creditos", "gerente"));

    options.AddPolicy("SoloLectura", p =>
        p.RequireRole("auditor", "consulta"));
});

// Proteger endpoints
[Authorize(Policy = "PuedeProcesarPagos")]
[HttpPost("transferencias")]
public async Task<IActionResult> Transfer(TransferirRequest req)
{ ... }

Principios de Seguridad por Diseño (Blueprint IBS)

🔍 Auditoría Inmutable

Toda operación sensible se registra en una tabla append-only. Nadie puede modificar el historial de operaciones. Requerimiento regulatorio bancario.

🔒 Segregación de Funciones

Quien aprueba no es quien ejecuta. Quien puede ver datos de clientes no puede procesar pagos. Previene fraude interno.

🛡️ DLP (Data Loss Prevention)

Datos sensibles (PII, números de cuenta) se tokenizan o cifran. Nadie accede al dato real salvo el módulo que lo gestiona.

🌐 Zero Trust en APIs

Toda llamada entre módulos debe autenticarse, incluso si vienen de dentro del mismo proceso. Un módulo comprometido no compromete a otros.
