Seguridad: Defensas en Profundidad

Más allá del Password: Identidad, Permisos y Cumplimiento.

En el IBS, adoptamos el modelo Zero Trust (Confianza Cero). Esto significa que no confiamos en nadie, esté dentro o fuera de la red del banco. Cada petición debe demostrar quién es y qué permiso tiene.

🔐

Autenticación (AuthN)

Identidad: "Soy el Cajero Juan".
Usamos OIDC. El sistema no guarda passwords; el Identity Provider (IdP) emite un token JWT firmado digitalmente que viaja en cada request.

🛡️

Autorización (AuthZ)

Capacidad: "¿Juan puede transferir $1M?".
Usamos Claims-Based Access Control. El token contiene "atributos" del usuario que el código valida antes de ejecutar lógica.

El JWT (JSON Web Token) bajo la lupa

En IBS, el token no es solo un string aleatorio; es un contenedor de información segura. Un desarrollador profesional debe saber leer sus partes:

{
"sub": "USR-99", // ID del usuario
"role": "Cajero",
"scope": "pagos.write", // Permiso técnico
"office": "Sucursal_001", // Claim de contexto
"exp": 1714000000 // Expiración (máx 15 min)
}

Implementación de Políticas en .NET 8

No usamos "If roles == 'admin'". Usamos Políticas de Seguridad descriptivas que separan la intención de la implementación.

// 1. Definir requerimientos complejos
builder.Services.AddAuthorization(options => {
    options.AddPolicy("DobleFactorRequerido", policy => 
        policy.RequireClaim("amr", "mfa")); // Solo si pasó 2FA

    options.AddPolicy("SoloHorarioBancario", policy =>
        policy.AddRequirements(new BusinessHoursRequirement()));
});

// 2. Aplicar de forma limpia
[Authorize(Policy = "DobleFactorRequerido")]
public async Task<IActionResult> AprobarPrestamo(...) { ... }

Seguridad de Datos y Auditoría (Regulación Bancaria)

Para un banco, el "quién hizo qué" es una prueba legal. Implementamos auditoría automática a nivel de infraestructura.

Amenaza Mitigación IBS Implementación
Fraude Interno Segregación de Funciones (SoD) Dos usuarios distintos para "Crear" y "Autorizar".
Inyección SQL ORM / Parametrización Uso mandatorio de EF Core (nunca concatenar strings).
Fuga de Datos Enmascaramiento (Masking) Los logs nunca muestran el PAN de la tarjeta.
Suplantación Tokens de corta duración JWTs expiran en 15 minutos + Refresh Tokens.
🚨
Principio de Menor Privilegio: Un desarrollador o un módulo debe tener solo los permisos mínimos necesarios para su función. Si el módulo de Notificaciones no necesita ver el saldo de la cuenta, el servicio de datos no debe exponérselo.
💡
Audit Log Inmutable: En IBS, los logs de auditoría se escriben en un almacenamiento que no permite borrar (WORM - Write Once Read Many). Esto garantiza que ni un administrador de base de datos pueda borrar las huellas de una operación fraudulenta.

Resumen Final de Seguridad:

La seguridad no es un obstáculo para el desarrollo, es la base de nuestro negocio. En el nuevo IBS, cada línea de código debe escribirse pensando: "¿Qué pasa si un atacante controla los parámetros de esta función?". Validar, Autenticar y Auditar son nuestras tres reglas de oro.
