DDD Práctico

Modelando el negocio bancario en el código, no en la base de datos.

DDD es un enfoque donde el Lenguaje Ubicuo manda. Si un analista bancario habla de "Sobregiro", el código debe tener una clase o método que diga Sobregiro. Pasamos de pensar en tablas a pensar en comportamientos.

Bloques de Construcción (Building Blocks)

Para nivelar el conocimiento, dividimos el dominio en estos conceptos fundamentales:

🏛️ Entity (Entidad)

Objetos que tienen una identidad que no cambia.
Ejemplo: Un Cliente. Aunque cambie su nombre, dirección o teléfono, sigue siendo el mismo cliente con el mismo ID.

💎 Value Object (Objeto Valor)

Objetos definidos por sus atributos. No tienen ID y son inmutables.
Ejemplo: Dinero(100, "USD"). Si cambias el 100 por 200, es otro objeto distinto. No "editas" el dinero, lo reemplazas.

🫙 Aggregate (Agregado)

Es una "frontera" de consistencia. Un grupo de objetos que se tratan como una unidad.
Ejemplo: CuentaCorriente es la raíz. Sus Movimientos pertenecen a ella. No puedes modificar un movimiento sin pasar por la Cuenta.

⚙️ Domain Service

Lógica de negocio que no cabe en una sola entidad.
Ejemplo: Un ConversorDivisas. No es responsabilidad de la Moneda ni de la Cuenta, es un servicio externo del dominio.

¿Dónde pongo mi código? (Diferencia de Servicios)

Esta es la duda más común. Aquí la regla clara para IBS:

Característica Application Service Domain Service
Responsabilidad Orquesta el flujo (Tarea técnica). Calcula reglas (Tarea de negocio).
Dependencias Llama a Repositorios, APIs, Mail. Solo usa Entidades y Value Objects.

Ejemplo en Código: Módulo de Cuentas

Fíjate cómo el código expresa reglas de negocio sin mencionar la base de datos:

// VALUE OBJECT: Inmutable y seguro
public record Saldo(decimal Monto, string Moneda) {
    public Saldo Sumar(Saldo otro) {
        if (Moneda != otro.Moneda) throw new Exception("Monedas distintas");
        return new Saldo(Monto + otro.Monto, Moneda);
    }
}

// ENTIDAD / AGGREGATE ROOT
public class CuentaCorriente {
    public Guid Id { get; }
    public Saldo SaldoActual { get; private set; }

    public void AplicarDebito(Saldo debito) {
        // La regla de negocio vive AQUÍ, no en un Store Procedure
        if (SaldoActual.Monto < debito.Monto) 
            throw new SaldoInsuficienteException();
        
        SaldoActual = new Saldo(SaldoActual.Monto - debito.Monto, SaldoActual.Moneda);
    }
}
💡
¿Por qué esto ayuda a IBS? Al sacar la lógica de los Stored Procedures y ponerla en Entidades de C#, podemos escribir Tests Unitarios. Podemos probar 1,000 escenarios de intereses o comisiones en 2 segundos sin siquiera prender la base de datos de Sybase.

📡 Domain Events: El pegamento del sistema

Cuando algo importante pasa en el dominio (ej: PrestamoAprobado), la entidad dispara un evento. Otros módulos pueden "escuchar" ese evento para, por ejemplo, enviar un mail o generar un asiento contable, sin que el módulo de Préstamos sepa quién lo escucha.
