Capas y Encapsulamiento

Blindando la lógica de negocio contra los cambios tecnológicos.

La Arquitectura "Cebolla" (Onion Architecture)

En el proyecto IBS, aplicamos una variante de Arquitectura Hexagonal. El principio fundamental es que la lógica de negocio (el Dominio) debe estar en el centro, aislada de las preocupaciones tecnológicas como bases de datos SQL o protocolos HTTP.

🖥️ Capa de Presentación (Entry Points) Web APIs, Controllers, SignalR Hubs, o CLI. Solo se encarga de recibir peticiones y devolver respuestas.
▼ invoca a ▼
⚙️ Capa de Aplicación (Orquestación) Use Cases (MediatR), Application Services. Orquesta la danza entre entidades y servicios externos.
▼ depende de ▼
💎 Capa de Dominio (El Corazón) Entities, Value Objects, Aggregates. Aquí vive el saber bancario (ej: cómo se calcula un interés).
▲ implementa interfaces de ▲
🗄️ Capa de Infraestructura (Detalles) EF Core Repositories, Redis Cache, Clientes de API externa. Es lo que permite que el sistema "respire".
⚠️
La Regla de Oro: Las dependencias siempre apuntan hacia adentro. El Dominio no depende de nada. La Infraestructura depende del Dominio. Si el Dominio necesita guardar un dato, define una interface y la Infraestructura la implementa.

Ejemplo Real: Módulo de Pagos

Observa cómo la lógica de validación de saldo es pura y no tiene referencias a bases de datos ni frameworks.

1. Dominio (Puro C#)

public class Transferencia {
  // Regla de Negocio: No se transfiere si no hay saldo
  public Result Procesar(Cuenta origen, Cuenta destino) {
    if (!origen.TieneSaldo(Monto)) 
        return Result.Fail("Saldo insuficiente");
    
    origen.Debitar(Monto);
    destino.Acreditar(Monto);
    return Result.Success();
  }
}

2. Aplicación (Orquestador)

public class TransferirHandler {
  private readonly ICuentaRepo _repo; // DIP en acción

  public async Task Handle(TransferirCmd cmd) {
    var origen = await _repo.GetById(cmd.From);
    var destino = await _repo.GetById(cmd.To);
    
    var txn = new Transferencia(cmd.Monto);
    var res = txn.Procesar(origen, destino);

    if(res.Ok) await _repo.UnitOfWork.Save();
  }
}

Beneficios para IBS 2026

🧩 Intercambiabilidad

Podemos cambiar Sybase por SQL Server o PostgreSQL mañana mismo, y el 100% de la lógica de Dominio quedará intacta.

🧪 Testability Total

Podemos testear las reglas de negocio en milisegundos sin necesidad de levantar una base de datos o un servidor web.

🛡️ Seguridad de Código

Evitamos que un desarrollador "accidentalmente" escriba lógica de negocio en un Stored Procedure o en un Controller.

💡
Arquitecto Senior dice: "La infraestructura es un detalle". Tu código debe ser capaz de explicar qué hace el banco (negocio) sin que yo tenga que saber qué base de datos usas.
