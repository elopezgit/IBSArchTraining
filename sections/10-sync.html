Integración Síncrona entre Módulos

Colaboración inmediata: El patrón Request/Response interno.

En el Monolito Modular de IBS, la integración síncrona ocurre cuando el Módulo A necesita una respuesta inmediata del Módulo B para decidir si continúa su proceso. Aunque ocurre dentro de la misma memoria (in-process), aplicamos el Principio de Menor Privilegio: el Módulo A solo ve la "cara pública" del Módulo B.

El Concepto de "Fachada de Módulo" (Module Facade)

Para mantener los módulos aislados, prohibimos que un módulo llame a las clases internas de otro. En su lugar, cada módulo expone un Contrato Público (Interface).

Secuencia: Validación de Disponibilidad de Fondos
[ Módulo PRÉSTAMOS ]          [ Módulo CUENTAS ]
         │                             │
         │─── ¿TieneSaldo(monto)? ────►│ (Ejecuta lógica de saldo)
         │       (Bloqueado)           │ (Consulta DB Schema Cuentas)
         │◄─────── true / false ───────│
         │                             │
    (Decide si                     (Libera hilo)
     desembolsa)

Implementación Profesional en .NET 8

Utilizamos el sistema de tipos de C# para forzar el encapsulamiento. El proyecto Contracts es lo único que otros módulos pueden "ver".

1. El Contrato (Módulo Cuentas)

// En IBS.Modules.Accounts.Shared
public interface IAccountsModule {
    Task<bool> HasEnoughFunds(
        Guid accountId, 
        decimal amount);
}

Este proyecto no tiene lógica, solo interfaces y DTOs.

2. El Consumidor (Módulo Préstamos)

public class LoanService(
    IAccountsModule accounts) {
    
    public async Task Process() {
        // Llamada síncrona:
        // Bloquea hasta recibir respuesta
        var ok = await accounts.HasEnoughFunds(...);
        if(!ok) throw new Exception();
    }
}

Análisis de Riesgos: El "Afecto Dominó"

Como profesionales, debemos saber que abusar de llamadas síncronas crea un sistema frágil. Si el Módulo B se vuelve lento, el Módulo A también se vuelve lento.

Escenario ¿Síncrono o Asíncrono? Razón Técnica
Validar si un DNI existe en Clientes. Síncrono Necesitas el dato YA para crear la cuenta.
Verificar saldo para un retiro. Síncrono Evita el doble gasto (Consistencia inmediata).
Notificar por Email un pago exitoso. Asíncrono Si el servidor de Mail falla, el pago debe seguir.
Generar el asiento contable. Asíncrono La contabilidad puede ocurrir milisegundos después.
💡
Regla de Oro en IBS: Si el flujo puede continuar aunque la otra parte falle o demore, usa Asincronismo (Eventos). Si el flujo NO PUEDE seguir sin la respuesta (porque es una validación crítica), usa Sincronismo (Fachadas).

Consideración de Performance (Threading)

En .NET 8, todas nuestras llamadas síncronas entre módulos DEBEN SER ASÍNCRONAS EN C# (usando Task y await). Esto evita el bloqueo de hilos (thread starvation) y permite que el servidor procese miles de transacciones concurrentes aunque los módulos se llamen entre sí.
