Límites entre Módulos

El concepto de Bounded Context: Donde las palabras cobran un único sentido.

La mayor causa de fracaso en la modernización de software no es el lenguaje de programación, sino los límites difusos. En DDD, un Bounded Context es una frontera lógica donde un modelo de datos es consistente y las palabras no tienen ambigüedad.

El Problema de la Ambigüedad: ¿Qué es un "Cliente"?

En el IBS legado, una tabla CLIENTE tiene 200 columnas porque intenta satisfacer a todos. En el Monolito Modular, el significado cambia según el contexto:

Módulo Clientes

Es un Sujeto con nombre, dirección y validación biométrica (KYC).

Módulo Créditos

Es un Deudor con capacidad de pago y score crediticio.

Módulo Contabilidad

Es una Contraparte vinculada a un asiento contable.

Estrategias para definir límites profesionales

1. Autonomía de Datos (Schema Isolation)

Cada módulo es dueño de sus tablas. Si el módulo de Pagos necesita el nombre del cliente, no hace un JOIN. O lo pide por servicio, o guarda una copia mínima (proyección) de lo que necesita.

2. Context Mapping

Es el mapa de cómo se hablan los módulos. Definimos quién es el "Upstream" (el que manda el dato) y quién es el "Downstream" (el que lo recibe).

Matriz de Responsabilidades IBS 2026

Para evitar solapamientos, hemos definido los límites de los módulos core basándonos en su comportamiento:

Módulo Responsabilidad Primaria "Verbo" Principal
Clientes Identidad y cumplimiento legal (KYC). Identificar
Cuentas Gestión de saldos y disponibilidad de fondos. Contener
Pagos Movimiento de dinero entre cuentas o bancos. Transferir
Créditos Ciclo de vida del préstamo y cobranza. Prestar
Contabilidad Registro de la verdad financiera (Libro Mayor). Registrar
⚠️
Regla de Oro contra el Acoplamiento: Si para hacer un cambio en Préstamos tienes que modificar el código de Contabilidad, tus límites están mal. La comunicación debe ser mediante Eventos: Préstamo dice "Me desembolsé", y Contabilidad reacciona creando el asiento.

Ejemplo Práctico: Comunicación entre Contextos

¿Cómo sabe el módulo de Pagos que el cliente está habilitado sin entrar a la base de datos de Clientes?

  1. 1. El usuario solicita un Pago.
  2. 2. Módulo Pagos llama a la IInternalClientService.GetStatus(id).
  3. 3. Esta interfaz está en un proyecto compartido, pero la implementación vive en el Módulo Clientes.
  4. 4. El Módulo Clientes responde "Habilitado".
  5. 5. Módulo Pagos procesa. (Ninguna tabla fue compartida en el proceso).
💡
Lección para el equipo: Los buenos límites hacen que el sistema sea fácil de entender. Si un módulo hace "un poco de todo", pronto será tan difícil de mantener como el IBS actual. Respetar el Bounded Context es proteger tu salud mental como desarrollador.
