Patrón Transactional Outbox

Garantizando la consistencia entre la base de datos y el mundo exterior.

En el IBS, no podemos permitir que un crédito se apruebe en la base de datos pero el módulo de Contabilidad nunca se entere porque el servidor de mensajes estaba caído. El Patrón Outbox asegura que ambos eventos ocurran, o ninguno ocurra.

El Desafío de la Doble Escritura (Dual Write)

Intentar escribir en dos sistemas distintos (SQL Server y RabbitMQ/MediatR) sin una transacción unificada es peligroso. El Patrón Outbox convierte una comunicación distribuida en una operación local atómica.

❌ El Riesgo: Falla Parcial

Si guardas el pago y luego intentas enviar el evento, un reinicio del servidor entre ambos pasos causará que el evento se pierda para siempre. Resultado: Dinero movido, pero no registrado contablemente.

✅ La Solución: Atocimidad SQL

Guardamos el "mensaje a enviar" en una tabla de la misma base de datos del módulo. Si la transacción SQL hace COMMIT, el mensaje está seguro. Si hace ROLLBACK, el mensaje desaparece con el error.

Flujo de Trabajo Profesional

Este patrón se divide en dos fases claramente separadas para garantizar que el usuario reciba su respuesta rápido sin esperar al Bus de eventos.

Arquitectura Outbox en IBS
[ Módulo Pagos ]
      │
      ▼ (Transacción Atómica)
┌──────────────────────────────┐
│  TABLE: Pagos (Monto: 100)   │
│  TABLE: Outbox (Msg: "Paid") │◄──┐
└──────────────────────────────┘   │
      │                            │ (Polleo o CDC)
      ▼                            │
[ Background Worker / Relay ] ─────┘
      │
      ▼ (Entrega Garantizada)
[ RabbitMQ / MediatR / ServiceBus ]

Implementación con .NET 8 y EF Core

Para simplificar el trabajo del equipo, automatizamos la captura de eventos de dominio durante el SaveChangesAsync.

// Interceptamos el guardado para llenar la tabla Outbox automáticamente
public override async Task<int> SaveChangesAsync(...) {
    var domainEvents = ChangeTracker.Entries<AggregateRoot>()
        .SelectMany(x => x.Entity.GetDomainEvents())
        .Select(domainEvent => new OutboxMessage {
            Id = Guid.NewGuid(),
            Type = domainEvent.GetType().Name,
            Content = JsonSerializer.Serialize(domainEvent),
            OccurredOnUtc = DateTime.UtcNow
        }).ToList();

    this.OutboxMessages.AddRange(domainEvents);
    return await base.SaveChangesAsync(cancellationToken);
}

Mecanismos de Relay (Publicación)

Una vez que el mensaje está en la tabla, necesitamos "empujarlo". Existen dos formas profesionales de hacerlo:

Método Descripción Uso en IBS
Polling Publisher Un Worker de .NET consulta la tabla cada 1 segundo buscando Processed = false. Recomendado (Simple y efectivo).
Transaction Log Tailing Herramientas como Debezium leen el log de la base de datos (CDC) sin afectar la performance. Solo para alta escala.
💡
At-Least-Once Delivery: Ten en cuenta que el Outbox garantiza que el mensaje se envíe al menos una vez. Debido a reintentos, el receptor podría recibirlo dos veces. Por eso, el Módulo que escucha (ej: Contabilidad) DEBE SER IDEMPOTENTE (usando el ID del mensaje original para ignorar duplicados).

Beneficios para la Migración:

Este patrón permite desacoplar la performance de Sybase. El usuario opera en la BD, y la integración pesada con otros sistemas ocurre en segundo plano, liberando los hilos de ejecución inmediatamente.
