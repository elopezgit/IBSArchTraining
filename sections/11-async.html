Integración Asíncrona — Colas y Eventos

Arquitectura dirigida por eventos (EDA): Desacoplando el tiempo y el espacio.

En el IBS moderno, la asincronía es nuestra herramienta para la Escalabilidad. El módulo emisor dispara un evento al viento (Event Bus) y se olvida. No le importa quién lo escucha ni cuánto tardan en procesarlo. Esto elimina el acoplamiento temporal.

Diferencias Críticas para el Diseño

📨 Comandos (Queues)

Intención: "Haz algo por mí".
Receptor: Uno solo (Point-to-point).
Ejemplo: GenerarResumenMensualCmd. Es una orden directa que debe ejecutarse una vez.

📡 Eventos (Pub/Sub)

Intención: "Algo ya pasó".
Receptor: Muchos (Broadcast).
Ejemplo: PagoRecibidoEvent. Notifica a Contabilidad, a la App Mobile y al área de Prevención de Fraude simultáneamente.

Garantía Bancaria: El Patrón Outbox

En un banco, no podemos permitir que un pago se guarde en la BD pero el evento de "Notificación" nunca se envíe porque el Event Bus estaba caído. Usamos la Tabla Outbox para garantizar la consistencia eventual.

  1. 1. En la misma transacción SQL se guarda el Pago y el Evento en una tabla Outbox.
  2. 2. Un proceso en segundo plano (Worker) lee la tabla y publica en el Bus.
  3. 3. Si el Bus falla, el mensaje sigue en la DB para reintentar. Cero pérdida de datos.

Implementación con MediatR y C# 12

Usamos records para los eventos por su inmutabilidad nativa y facilidad de transporte.

// 1. Definición del Evento (Dato puro e inmutable)
public record PagoExitosoEvent(Guid PagoId, decimal Monto) : INotification;

// 2. Publicador (En el Módulo de Pagos)
public class PagoService(IMediator bus) {
    public async Task Finalizar(Guid id) {
        // ... lógica de persistencia
        await bus.Publish(new PagoExitosoEvent(id, 1500.00m));
    }
}

// 3. Suscriptor (En el Módulo de Notificaciones)
public class NotificarUsuarioHandler : INotificationHandler<PagoExitosoEvent> {
    public async Task Handle(PagoExitosoEvent ev, CancellationToken ct) {
        // Enviar Push al celular del cliente
        await _pushService.Send($"Recibiste un pago de {ev.Monto}");
    }
}

Cuándo elegir Asincronismo (The Trade-off)

Ventaja Desventaja
Resiliencia: Si falla el módulo de mails, el pago no se cae. Complejidad: Más difícil de seguir el flujo (Debugging).
Performance: El usuario recibe el OK más rápido. Consistencia: Los datos tardan milisegundos en replicarse.
💡
Concepto de Nivelación: Consistencia Eventual. A diferencia de la consistencia inmediata (síncrona), aquí aceptamos que el saldo se actualice YA, pero que el mail llegue 2 segundos después. En el 90% de los procesos bancarios (excepto el movimiento de dinero core), la consistencia eventual es suficiente y preferible.

Resumen para el equipo:

Síncrono para leer datos críticos o validar. Asíncrono para reaccionar a cambios o ejecutar procesos pesados en segundo plano. Esta es la clave para que el IBS no se bloquee bajo mucha carga.
