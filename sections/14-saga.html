Patrón SAGA

Gestionando transacciones distribuidas con integridad bancaria.

En el monolito modular, cada módulo tiene su propio "límite de consistencia". No podemos hacer un BEGIN TRANSACTION que abarque Clientes, Cuentas y Contabilidad al mismo tiempo. La SAGA es una secuencia de transacciones locales coordinadas que garantiza la Consistencia Eventual.

El Concepto: Acción y Compensación

Cada paso de una SAGA tiene una "acción" (hacer algo) y una "compensación" (deshacerlo si algo sale mal más adelante). Es como reservar un hotel: si el vuelo se cancela, debes cancelar la reserva del hotel para no perder dinero.

🎼 Orquestación (Centralizado)

Un "Cerebro" central dice a cada módulo qué hacer. Es el modelo que usamos en IBS por su facilidad de auditoría.

  • Ideal para procesos complejos.
  • Fácil de visualizar el estado del proceso.
  • Control total sobre el flujo.

💃 Coreografía (Descentralizado)

Los módulos se pasan "la posta" mediante eventos. No hay un jefe central.

  • Más desacoplado.
  • Difícil de debuggear en sistemas bancarios.
  • Riesgo de dependencias cíclicas.

Anatomía de una Transferencia en IBS

Observemos cómo se comporta el sistema ante un fallo en el último paso (Contabilidad):

Paso Módulo Acción Positiva Acción Compensatoria
1 Cuentas Reservar Saldo (Estado: Pendiente) Liberar Reserva
2 Pagos Registrar Orden de Pago Marcar Pago como Fallido
3 Contabilidad Generar Asiento GL Generar Asiento de Reversa

Implementación Profesional: State Machines

En .NET 8, no escribimos try-catch gigantes. Usamos Máquinas de Estado que persisten el progreso en la base de datos para que, si el servidor se apaga, la SAGA pueda retomar donde quedó.

// Ejemplo conceptual de Orquestador persistente
public class TransferenciaSagaState {
    public Guid CorrelationId { get; set; }
    public string CurrentState { get; set; } // "SaldoReservado", "PagoRegistrado"
    public decimal Monto { get; set; }
}

// Lógica de compensación automática
if (stepFailed) {
    await SendCompensatingCommands(sagaState.CorrelationId);
}
⚠️
El Peligro del "Zombi": Sin una SAGA, si el paso 3 falla pero el paso 1 ya debitó el dinero, el cliente pierde sus fondos y el banco queda descuadrado. La SAGA asegura que el dinero siempre tenga un destino final válido o regrese a su origen.
💡
Nivelación Pro: Las SAGAs requieren que cada servicio sea Idempotente. Si la orquestación reintenta un paso de compensación, el módulo de Cuentas no debe liberar el saldo dos veces.

Resumen para el equipo:

La SAGA es lo que convierte una serie de pasos aislados en un Proceso de Negocio Robusto. En el nuevo IBS, cada vez que una operación involucre más de un módulo, diseñaremos una SAGA con sus respectivas compensaciones.
