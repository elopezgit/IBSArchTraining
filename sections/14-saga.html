Patrón SAGA

Consistencia en transacciones que tocan múltiples módulos

El Problema: Transacciones Cross-Módulo

En IBS, una transferencia toca Pagos, Cuentas y Contabilidad. No podemos usar una sola transacción ACID porque cada módulo tiene su propia BD (o schema). Necesitamos coordinar una secuencia de transacciones locales con capacidad de deshacer los pasos anteriores si algo falla.

¿Qué es una SAGA?

Una secuencia de transacciones locales, donde cada paso publica un evento/mensaje que dispara el siguiente. Si un paso falla, se ejecutan transacciones compensatorias para deshacer los pasos anteriores.

SAGA: Flujo de Transferencia — Camino Feliz
  API Gateway
      │
      ▼
  ┌─────────────────────────────────────────────────────────────────┐
  │                  SAGA ORQUESTADOR                               │
  │                                                                 │
  │  PASO 1: Validar Cliente (Módulo Clientes)                      │
  │     ──────────────────────────────► ok                         │
  │                                                                 │
  │  PASO 2: Reservar Saldo (Módulo Cuentas)                        │
  │     ──────────────────────────────► ok (saldo reservado)       │
  │                                                                 │
  │  PASO 3: Registrar Pago (Módulo Pagos)                          │
  │     ──────────────────────────────► ok                         │
  │                                                                 │
  │  PASO 4: Postear GL (Módulo Contabilidad)                       │
  │     ──────────────────────────────► ok                         │
  │                                                                 │
  │  PASO 5: Confirmar Saldo (Módulo Cuentas)                       │
  │     ──────────────────────────────► ok ✅                      │
  └─────────────────────────────────────────────────────────────────┘
SAGA: Flujo con Falla — Compensaciones
  PASO 1: Validar Cliente ──────────────────► ok
  PASO 2: Reservar Saldo  ──────────────────► ok (saldo reservado)
  PASO 3: Registrar Pago  ──────────────────► ok
  PASO 4: Postear GL      ──────── FALLA ❌

  COMPENSACIONES (en orden inverso):
  COMP-3: Cancelar Pago   ──────────────────► ok
  COMP-2: Liberar Saldo   ──────────────────► ok (saldo liberado)
  COMP-1: (no necesaria)
  
  Estado final: Todo revertido, cliente notificado del fallo

Orquestación vs Coreografía

🎼 Orquestación (elegido IBS)

Un componente central (Saga Orchestrator) coordina los pasos y toma decisiones. Más fácil de entender, testear y monitorear.

SagaOrchestrator
  ├── paso1 → Clientes
  ├── paso2 → Cuentas
  └── paso3 → Pagos

💃 Coreografía

Cada servicio reacciona a eventos de los demás. No hay coordinador central. Más flexible pero más difícil de trazar y debuggear.

Cuentas emite → Pagos escucha
Pagos emite → Contabilidad escucha
(emerge la coordinación)

Implementación básica en .NET

public class TransferenciaSaga
{
    private readonly IClientesFacade _clientes;
    private readonly ICuentasFacade _cuentas;
    private readonly IPagosFacade _pagos;
    private readonly ISagaStateRepository _state;

    public async Task<SagaResult> Execute(TransferenciaCmd cmd)
    {
        var sagaId = Guid.NewGuid();
        await _state.Save(sagaId, "Iniciada");

        // PASO 1
        var kycOk = await _clientes.ValidarKyc(cmd.ClienteId);
        if (!kycOk) return SagaResult.Failure("KYC inválido");

        // PASO 2
        var reserva = await _cuentas.ReservarSaldo(cmd.CuentaOrigen, cmd.Monto);
        if (!reserva.Ok) return SagaResult.Failure("Saldo insuficiente");

        try
        {
            // PASO 3
            await _pagos.RegistrarPago(cmd, reserva.Id);
            // PASO 4 (postear GL via evento async)
            // PASO 5
            await _cuentas.ConfirmarDebito(reserva.Id);
            await _state.Save(sagaId, "Completada");
            return SagaResult.Success();
        }
        catch (Exception ex)
        {
            // COMPENSACIONES
            await _pagos.CancelarPago(reserva.Id);
            await _cuentas.LiberarReserva(reserva.Id);
            await _state.Save(sagaId, "Compensada");
            return SagaResult.Failure(ex.Message);
        }
    }
}
⚠️
Requisito crítico: Cada paso de la SAGA debe ser idempotente. Si el proceso falla a mitad de una compensación, debe poder reintentarse sin duplicar efectos. Por eso se guardan los IDs de reserva y se usa el patrón Outbox para los eventos que disparan cada paso.
