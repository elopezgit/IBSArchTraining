Atributos de Calidad

Las "-ilities": el verdadero trabajo del arquitecto

En una arquitectura bancaria, los atributos de calidad no son detalles secundarios ni deseos generales. Son restricciones reales de diseño que condicionan cada decisión técnica: cómo se separan los módulos, cómo se exponen las APIs, cómo se protege la información, cómo se audita una operación y cómo se responde ante fallas.

Una funcionalidad puede estar correctamente programada desde el punto de vista del negocio, pero si no cumple con seguridad, trazabilidad, disponibilidad, mantenibilidad y performance, puede transformarse en un riesgo operativo. Por eso, en IBS, la arquitectura no se evalúa solamente preguntando: “¿funciona?”, sino también: “¿funciona de forma segura, mantenible, auditable y escalable?”

Idea central

Los atributos de calidad son las características que definen qué tan confiable, seguro, mantenible y preparado está un sistema para operar en escenarios reales. En sistemas bancarios, muchas veces estos atributos pesan más que la propia funcionalidad visible para el usuario.

Principales atributos priorizados para IBS

⚡

Performance

No significa simplemente que el sistema sea “rápido”. En arquitectura se analiza mediante métricas concretas: latencia, tiempo de respuesta, throughput y consumo de recursos.

En .NET, esto implica cuidar consultas a base de datos, llamadas HTTP, serialización JSON, uso de memoria, logs excesivos y operaciones bloqueantes.

Ejemplo: p95 menor a 500ms
🔄

Disponibilidad

Es la capacidad del sistema de seguir operando incluso cuando algo falla. No se trata de evitar todas las fallas, sino de diseñar el sistema para tolerarlas.

En .NET puede trabajarse con health checks, retry policies, circuit breakers, balanceo de carga, monitoreo y despliegues controlados.

Objetivo: reducir indisponibilidad
📈

Escalabilidad

Es la capacidad del sistema para soportar crecimiento sostenido de usuarios, operaciones o volumen de datos sin degradarse de forma crítica.

En un monolito modular con .NET, se puede escalar verticalmente aumentando recursos del servidor, u horizontalmente desplegando más instancias detrás de un balanceador.

Clave: diseño modular
🔒

Seguridad y Auditoría

En sistemas bancarios, la seguridad no puede agregarse al final. Debe estar presente desde el diseño: autenticación, autorización, validación de datos, trazabilidad, cifrado y control de accesos.

Cada operación sensible debe dejar una huella auditable: quién la hizo, cuándo, desde dónde, con qué datos y cuál fue el resultado.

Prioridad crítica
🔧

Mantenibilidad

Un sistema mantenible permite modificar, extender y corregir funcionalidades sin romper otras partes. Esto depende del bajo acoplamiento, alta cohesión, separación de responsabilidades y claridad del código.

En .NET se refleja en una buena separación entre Controllers, Services, Repositories, DTOs, validadores, clientes externos y capa de dominio.

Menos deuda técnica
🧪

Testability

Es la facilidad para probar el sistema de forma automática o controlada. En una modernización bancaria, la testabilidad reduce el riesgo de regresiones.

En .NET conviene diseñar servicios desacoplados mediante interfaces, inyección de dependencias, pruebas unitarias, pruebas de integración y mocks para APIs externas.

Clave para migraciones

Ejemplo aplicado en .NET

Supongamos una API bancaria desarrollada en .NET que expone una operación transaccional. El objetivo no es solamente que el endpoint responda correctamente, sino que cumpla varios atributos al mismo tiempo.

Endpoint funcional pero débil

[HttpPost("operacion")]
public IActionResult RegistrarOperacion(OperacionRequest request)
{
    var resultado = _service.Procesar(request);
    return Ok(resultado);
}

Este código puede funcionar, pero todavía no expresa claramente validaciones, auditoría, manejo de errores, trazabilidad, seguridad ni resiliencia.

Endpoint más alineado a calidad

[HttpPost("operacion")]
public async Task<IActionResult> RegistrarOperacion(
    [FromBody] OperacionRequest request,
    CancellationToken cancellationToken)
{
    var correlationId = HttpContext.TraceIdentifier;

    _logger.LogInformation(
        "Inicio operación bancaria. CorrelationId: {CorrelationId}",
        correlationId);

    var resultado = await _operacionService.ProcesarAsync(
        request,
        correlationId,
        cancellationToken);

    return Ok(resultado);
}

Esta versión ya incorpora asincronismo, trazabilidad, logging, cancellation token y separación de responsabilidades hacia una capa de servicio.

Escalabilidad vs Elasticidad

Es importante no confundir estos conceptos. Ambos están relacionados con carga, pero responden a problemas distintos.

📊 Escalabilidad

Es la capacidad de crecer de manera planificada. Por ejemplo, si el banco pasa de procesar miles a millones de operaciones mensuales, el sistema debe poder acompañar ese crecimiento.

Se resuelve con buena arquitectura, separación de módulos, optimización de base de datos, diseño de APIs y capacidad de desplegar más instancias.

Crecimiento sostenido

⚡ Elasticidad

Es la capacidad de aumentar o reducir recursos dinámicamente según la demanda. Por ejemplo, más instancias durante cierre de mes o picos de pagos.

Se relaciona más con infraestructura, contenedores, cloud, balanceadores, monitoreo y reglas automáticas de escalado.

La matriz de conflictos

En arquitectura no existe una solución perfecta. Cada mejora suele tener un costo. El trabajo del arquitecto consiste en identificar esos costos y decidir conscientemente.

Decisión Técnica Atributo Mejorado Atributo Afectado Explicación
Cifrar información sensible Seguridad Performance El cifrado agrega procesamiento adicional en lectura y escritura.
Agregar caché Performance Consistencia Puede devolver información desactualizada si no se invalida correctamente.
Separar módulos Mantenibilidad Complejidad Mejora el orden, pero exige más disciplina en contratos e integración.
Agregar auditoría detallada Trazabilidad Volumen de datos Genera más registros y requiere políticas de almacenamiento.
Usar múltiples instancias Disponibilidad Operación Requiere balanceo, monitoreo y control de estado distribuido.

Buenas prácticas concretas para .NET

Para Performance

  • Usar métodos async/await en operaciones de I/O.
  • Evitar consultas innecesarias a base de datos.
  • Paginar respuestas grandes.
  • Medir tiempos con logs y métricas.
  • Evitar traer más columnas de las necesarias.

Para Seguridad

  • Validar siempre los datos de entrada.
  • No exponer errores internos al cliente.
  • Usar JWT, OAuth o mecanismos definidos por arquitectura.
  • Aplicar autorización por rol, tarea o facultad.
  • Registrar operaciones sensibles en auditoría.

Para Disponibilidad

  • Implementar health checks.
  • Usar retries controlados para servicios externos.
  • Aplicar circuit breaker ante fallas repetidas.
  • Diseñar respuestas claras ante indisponibilidad parcial.
  • Evitar que una falla externa bloquee todo el flujo.

Para Mantenibilidad

  • Separar Controllers, Services, Repositories y DTOs.
  • No colocar lógica de negocio dentro del Controller.
  • Usar nombres claros y consistentes.
  • Centralizar validaciones repetidas.
  • Evitar métodos extensos con demasiadas responsabilidades.
💡
Veredicto IBS: en un sistema bancario, la jerarquía de atributos debe ser clara: 1. Seguridad y Consistencia, 2. Auditoría y Trazabilidad, 3. Disponibilidad, 4. Performance. Es preferible una operación 100ms más lenta antes que una operación rápida sin trazabilidad, sin control de permisos o con riesgo de inconsistencia financiera.

Conclusión arquitectónica

Los atributos de calidad permiten transformar una solución técnica en una arquitectura confiable. En IBS, cada API, módulo, integración o proceso batch debería poder responder estas preguntas:

  • ¿Es seguro?
  • ¿Es auditable?
  • ¿Es mantenible?
  • ¿Puede escalar?
  • ¿Tolera fallas?
  • ¿Se puede probar sin depender de todo el sistema?

Si una solución no puede responder estas preguntas, todavía no está lista desde el punto de vista arquitectónico.
