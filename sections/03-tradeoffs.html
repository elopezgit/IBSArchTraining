Trade-offs Arquitectónicos

"Everything in software architecture is a trade-off." — Mark Richards

La Primera Ley de la Arquitectura

Si alguien cree haber encontrado una solución que no tiene desventajas, es muy probable que aún no las haya identificado. No existen las "mejores prácticas", solo existen los trade-offs.

Ejemplo Core: ¿SAGAs o Transacciones Distribuidas?

En la modernización de IBS, la integridad de los datos es sagrada. Pero cómo la logramos define la agilidad del sistema.

Elegido para IBS

✅ SAGA (Coreografía/Orquestación)

Ventajas: Alta disponibilidad, bajo acoplamiento entre módulos, el sistema no se detiene si un servicio falla.

Costos: Consistencia eventual (el saldo puede tardar ms en actualizarse), requiere lógica de compensación (rollback manual).

Descartado por Escala

❌ 2PC (Two-Phase Commit)

Ventajas: Consistencia ACID garantizada de forma nativa, programar es más sencillo (parece una DB única).

Costos: "Protocolo del suspenso": si un nodo falla, todos se bloquean. Muy baja performance en alta carga.

Registro de Decisión (ADR)

Toda decisión relevante debe quedar documentada. Esto evita discusiones futuras y permite entender el "por qué" de la arquitectura.

Ejemplo ADR: Uso de SAGA

ADR-003: Uso de SAGA para transacciones distribuidas

Estado: Aceptado

Contexto:
El sistema requiere alta disponibilidad y desacoplamiento entre servicios.

Opciones evaluadas:
- SAGA
- 2PC

Decisión:
Se adopta SAGA.

Trade-offs:
+ Mayor resiliencia
+ Mejor escalabilidad
- Consistencia eventual
- Complejidad en compensaciones

Consecuencias:
Se implementarán mecanismos de compensación y monitoreo.

Análisis ATAM (Atributos de Calidad)

Evaluamos cada decisión en función de atributos de calidad del sistema.
