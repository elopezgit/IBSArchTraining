Arquitectura de Monolito Modular

Diseñando para la autonomía de equipos y la integridad bancaria.

Definición y Filosofía

El Monolito Modular (o Modulith) es un estilo arquitectónico que equilibra la simplicidad operativa de una sola unidad de despliegue con el rigor de separación de intereses de los microservicios. En el proyecto IBS, esto significa que el sistema "grita" su intención de negocio a través de su estructura de carpetas.

Flujo de Comunicación en el Monolito Modular
┌──────────────────────────────────────────────────────────┐
│             HOST APPLICATION (.NET 8 Web API)            │
├──────────────────────────────────────────────────────────┤
│  ┌──────────────┐      ┌──────────────┐      ┌────────┐  │
│  │   Módulo     │      │   Módulo     │      │ Módulo │  │
│  │  Clientes    │◄────►│   Pagos      │◄────►│ Contab.│  │

          ▼                     ▼                   ▼

│ Schema: Customers │ │ Schema: Payments │ │ Schema: Ledger │
└───────────────────┘ └──────────────────┘ └────────────────┘

Cada módulo posee sus datos (Schemas) y se comunica mediante contratos o eventos internos.

Los Pilares de la Independencia

📦 Aislamiento de Datos

Teoría: La base de datos es el mayor punto de acoplamiento.
Ejemplo IBS: El módulo de Préstamos no puede hacer un JOIN con la tabla de Clientes. Debe consultar la API interna del módulo o tener una proyección de datos local.

🔗 Comunicación Inter-Módulo

Teoría: Inversión de Dependencias (DIP).
Ejemplo IBS: Usamos un In-Memory Event Bus (MediatR) para notificar cambios. "Pago Realizado" dispara "Generar Asiento Contable" sin que el módulo de pagos conozca al contable.

🚪 Public API / Contract

Teoría: Ocultamiento de información.
Ejemplo IBS: Solo el proyecto IBS.Modules.X.Shared es visible para otros módulos. Las implementaciones internas en Infrastructure son privadas (internal en C#).

🌱 Límites de Transacción

Teoría: Consistencia Atómica.
Ejemplo IBS: Una transacción SQL solo puede afectar tablas del mismo módulo. Para operaciones cross-module, usamos el patrón Outbox.

Organización del Código: Capas vs. Módulos

Adoptamos la Arquitectura Hexagonal dentro de cada módulo para asegurar que la lógica de negocio bancaria sea testeable e independiente de la infraestructura.

🗂️ Por Capa (Antipatrón)

src/
  Controllers/
  Services/
  Entities/

El "Big Ball of Mud". Cambiar un campo en Clientes requiere tocar todas las carpetas del proyecto.

🧩 Estructura IBS (.NET)

src/Modules/Payments/
  /Domain    (Reglas)
  /Application (Uso)
  /Infrastructure (DB)
  /API        (Web)

"Screaming Architecture": Al ver la carpeta, entiendes que el sistema gestiona Pagos, no carpetas técnicas.

🧪 Testability

tests/
  /Payments.Unit
  /Payments.Integ
  /ArchTests

Cada módulo se testea de forma aislada, garantizando que el Core Bancario sea robusto.

💡
Arquitectura Evolutiva: El Monolito Modular nos permite empezar con una infraestructura simple. Si el día de mañana el volumen de Tarjetas de Crédito crece exponencialmente, podemos extraer esa carpeta y convertirla en un Microservicio físico con un esfuerzo mínimo de refactorización.

Ejemplo de Implementación: El "Muro de Contención"

Para evitar que el código se degrade, usamos NetArchTest en el CI/CD:

// Regla de Arquitectura:
Types.InAssembly(paymentsAssembly)
     .ShouldNot()
     .HaveDependencyOn("IBS.Modules.Accounting.Infrastructure")
     .Check();

Esto garantiza que el acoplamiento se mantenga en cero mediante pruebas automatizadas.
