Inyección de Dependencias (DI)

El arte de no fabricar tus propias herramientas.

Para entender la DI, usemos una analogía: Si eres un electricista (una clase de negocio), tú no fabricas tu propio destornillador (dependencia). Tú esperas que el destornillador esté en tu caja de herramientas cuando llegas a la obra. La DI es quien te entrega la herramienta.

¿Qué problema resolvemos?

El mayor enemigo de un sistema bancario que debe durar 20 años es el Acoplamiento Rígido. Si una clase crea a otra usando new(), queda "pegada" a ella para siempre.

❌ Sin DI (Control Centralizado)

La clase decide qué herramienta usar y cómo crearla.

  • No se puede testear sin la base de datos real.
  • Si cambias SQL por Oracle, rompes 50 clases.
  • Código "frágil" y difícil de leer.

✅ Con DI (Inversión de Control)

La clase solo pide qué necesita (interfaz), no le importa quién se lo da.

  • Testeo simple usando "Mocks" (herramientas de juguete).
  • Cambios tecnológicos transparentes.
  • Cumplimos el principio SOLID de Inversión de Dependencias.

Implementación Profesional en IBS

En .NET 8, el contenedor de dependencias es el motor del sistema. Sigue este flujo de 3 pasos:

1. La Abstracción (El Contrato)

public interface IRepositorioCuentas {
    Task<Cuenta> ObtenerPorId(int id);
}

Ubicación: Capa de Dominio. No sabe nada de SQL.

2. El Servicio (El Consumidor)

public class ProcesadorPagos(IRepositorioCuentas repo) {
    // Usamos el "Primary Constructor" de C# 12
    public async Task Procesar(int id) {
        var cuenta = await repo.ObtenerPorId(id);
    }
}

Ciclos de Vida (Lifetimes) en .NET

Es fundamental entender cuánto tiempo vive un objeto inyectado para evitar errores de memoria o datos corruptos:

Lifetime Comportamiento Uso común en IBS
Transient Se crea una instancia cada vez que se pide. Servicios ligeros sin estado.
Scoped Se crea una instancia por cada petición HTTP. Contextos de Base de Datos (DbContext) y Repositorios.
Singleton Se crea una sola vez y vive siempre. Cachés compartidas, configuración de sistema.
💡
Estrategia de Migración: Gracias a la DI, podemos implementar una interfaz llamada IServicioContable que por debajo llame a un Stored Procedure de Sybase hoy, y mañana, simplemente cambiando una línea en Program.cs, que llame a un nuevo servicio en C#/.NET. El resto del sistema ni se enterará del cambio.

Resumen para el equipo:

No uses new para clases que tengan lógica de negocio o acceso a datos. Pídelas en el constructor. Deja que el contenedor de .NET haga el trabajo sucio por ti.

Inyección de Dependencias y Capas

Es el pegamento que permite la independencia entre la lógica de negocio y la tecnología[cite: 13]. Cada módulo se organiza en capas (Onion Architecture)[cite: 10]:

  • Domain: El corazón del módulo; contiene entidades y reglas de negocio sin dependencias externas[cite: 17, 18].
  • Application: Orquestador de casos de uso; aquí se definen las interfaces[cite: 12, 18].
  • Infrastructure: Implementaciones técnicas (Acceso a Sybase, Clientes API, Colas)[cite: 10].

Beneficio: La capa de Aplicación pide una interfaz (ej. ICuentaRepository) y .NET 8 inyecta la implementación de Infraestructura, permitiendo cambiar la base de datos sin tocar la lógica de negocio[cite: 13].
