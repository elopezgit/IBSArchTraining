Inyección de Dependencias

El mecanismo que hace posible el bajo acoplamiento

La Inyección de Dependencias (DI) es el patrón que permite que las clases reciban sus dependencias desde afuera en lugar de crearlas internamente. Es el mecanismo central del Principio de Inversión de Dependencias (la "D" de SOLID).

Problema sin DI (acoplamiento duro)

public class PagoService
{
    // ❌ PagoService crea sus dependencias — acoplamiento duro
    private readonly SqlCuentaRepository _repo = new SqlCuentaRepository();

    // No podemos testear sin BD real
    // No podemos cambiar la implementación sin tocar PagoService
}

Solución con DI (inversión de dependencias)

// 1. Definir la abstracción (en capa de Dominio)
public interface ICuentaRepository
{
    Task<Cuenta> GetById(CuentaId id);
    Task Save(Cuenta cuenta);
}

// 2. Implementación en Infraestructura
public class EfCuentaRepository : ICuentaRepository
{
    private readonly IbsDbContext _db;
    public EfCuentaRepository(IbsDbContext db) => _db = db;
    public async Task<Cuenta> GetById(CuentaId id)
        => await _db.Cuentas.FindAsync(id.Value);
}

// 3. PagoService recibe la abstracción — no sabe qué implementación
public class PagoService
{
    private readonly ICuentaRepository _repo;  // ✅ interfaz
    public PagoService(ICuentaRepository repo) => _repo = repo;
}

// 4. Registrar en el contenedor DI (Program.cs)
builder.Services.AddScoped<ICuentaRepository, EfCuentaRepository>();
// Para tests: AddScoped<ICuentaRepository, InMemoryCuentaRepository>()
💡
Beneficio clave para IBS: La migración de stored procedures a .NET se hace módulo por módulo. Con DI, el módulo "Pagos" puede tener una implementación temporal que llama al SP (SqlPaymentRepository) y luego reemplazarla por la implementación .NET pura (EfPaymentRepository), sin tocar el código de dominio ni aplicación.
